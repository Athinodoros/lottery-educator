## Output Task Documents

You will generate THREE synchronized task documents. Each document provides a different perspective on the same project tasks and must be kept in sync:

### 1. Phased Roadmap - What to build in order

**Purpose**: Sequential development phases with clear completion criteria and dependencies

**Structure**: 
- **Phase 0**: Project Setup & Infrastructure
- **Phase 1**: MVP Backend Services Core
- **Phase 2**: Frontend Setup & Core Features
- **Phase 3**: Testing & Quality Assurance
- **Phase 4**: Logging, Monitoring & DevOps
- **Phase 5**: Polish, Performance & Optional Features
- **Phase 6**: Deployment & Production Hardening

**Key Requirement**: Each phase must have clear completion criteria and explicit dependencies on previous phases

### 2. Architectural Layers - Organize by concern

**Purpose**: Group tasks by architectural layer to maintain clean separation of concerns and logical responsibility

**Structure**:
- **Infrastructure & Platform**: Docker, Kubernetes, Docker Compose setup
- **Backend Services**: Define each microservice with its responsibilities
  - API Gateway/Load Balancer
  - Core Business Logic Services
  - Data Access & Database Layer
  - Message Queues/Event Bus (if applicable)
- **Frontend Layer**: UI components, state management, API integration
- **Testing Infrastructure**: Test frameworks, fixtures, test data, CI/CD testing pipelines
- **Observability Stack**: Logging, metrics collection, alerting, dashboards
- **Documentation**: Architecture diagrams, API docs, runbooks

### 3. Feature-Based Breakdown - What to implement

**Purpose**: Feature-centric view showing implementation details for each core capability

**For Each Core Feature/Module**:
- Which microservice(s) handle it
- Frontend component requirements
- Database schema/models
- API contracts
- Tests needed (unit, integration, E2E)
- Logging & monitoring points
- Deployment considerations

## Task Generation Standards

Apply these standards consistently to all generated tasks:

1. **Task Atomicity**: Each task must be completable in 1-3 days by a single developer

2. **Task Dependencies**: Include explicit dependencies between tasks using format "Depends on: [Task ID]"

3. **Required Skills**: Specify required skills for each task (e.g., "Frontend", "Backend TypeScript", "DevOps", "Database Design")

4. **Acceptance Criteria**: Define clear acceptance criteria—what must be delivered to mark task complete

5. **Risk & Complexity**: Call out tasks needing extra review or pairing; flag complexity levels

6. **Testing Integration**: Each backend/frontend task must state what tests are needed (unit, integration, E2E scope)

7. **Logging & Monitoring**: Every service task must include logging instrumentation and metrics export

8. **RALF Loop Integration**: Structure each task to support the RALF loop:
   - **Retrieval**: What information/resources must be gathered? (architecture docs, API specs, data models, dependencies)
   - **Analysis**: How should requirements be analyzed? (acceptance criteria breakdown, technical feasibility assessment, impact analysis)
   - **Logical Reasoning**: What is the technical approach? (architecture decisions, tech choices, design patterns)
   - **Feedback**: How is progress validated? (code review checkpoints, testing stages, integration points, stakeholder sign-off)

9. **Document Synchronization**: 
   - When a phase includes a feature, reference it in the Layers section
   - When implementing a layer component, cross-reference which phase and features depend on it
   - Maintain consistent task IDs across all three documents

## Project Structure & Documentation Standards

**Critical Instruction for Developers (and AI Agents)**:

As the project grows and new directories/modules are created, maintain a `README.md` file in each directory that provides:

1. **Purpose**: What does this directory/module do?
2. **Contents**: Quick overview of files and subdirectories
3. **Key Concepts**: Domain-specific terminology and patterns used
4. **Dependencies**: What other modules/services does this depend on?
5. **Quick Start**: How to set up or work with this module
6. **Important Notes**: Any gotchas, conventions, or critical information

**Why This Matters**:
- Speeds up agent/LLM understanding without requiring full file reads
- Helps new developers (or returning developers) get context quickly
- Reduces context window usage in LLM-assisted development
- Creates a living documentation that evolves with the codebase
- Essential for maintaining code quality as the project scales

**Example Structure**:
```
project-root/
├── README.md (project overview)
├── backend/
│   ├── README.md (backend architecture)
│   ├── services/
│   │   ├── service-name/
│   │   │   └── README.md (service-specific documentation)
│   │   └── ...
│   └── ...
├── frontend/
│   ├── README.md (frontend structure)
│   ├── components/
│   │   └── README.md (component conventions)
│   └── ...
└── database/
    └── README.md (schema overview)
```
